4 Etiketleme mantığı — neden null, false değildir?

Kanıt yoksa iddia yok

has_elevator = true → asansör kanıtlandı
has_elevator = false → açıkça “yok” denildi
has_elevator = null → bilmiyoruz
  • Checkbox seçili değilse otomatik false yazmıyoruz
  • Fotoğrafta görünmeyen özellik “yok” sayılmaz
  • Kullanıcı “asansörlü” derse yalnız kanıtlanmış true ilanlar geçer
Monotonik merge
1. Scraper explicit fact'i yazar.
2. DeepSeek yalnız halen null metin alanlarını doldurur.
3. Kimi yalnız halen null görsel alanlarını doldurur.

Sonraki kaynak önceki kanıtı ezemez.
filter_sources["has_elevator"] = "scraper_property_feature"
Bu slayt sistemin güven mantığı. False ile null ayrımı yapılmazsa sistem sessizce yanlış eleme yapar. Ayrıca provenance sayesinde her fact'in nereden geldiğini gösterebiliriz.

5 Retrieval nasıl çalışıyor? — matematik sezgisi

Önce hızlı aday bul, sonra pahalı biçimde sırala

cosine(q, d) = (q · d) / (‖q‖ ‖d‖)
yüksek skor → sorgu ve ilan vektörleri benzer yönde
  • BGE-M3 bi-encoder: sorgu ve ilan ayrı kodlanır
  • İlan vektörleri önceden hesaplanır; sorguda tekrar modelden geçmez
rerank score = CE([query ; listing])
birlikte okur → daha doğru, daha pahalı
  • Cross-encoder yalnız daraltılmış top-N adayda çalışır
  • HNSW tüm vektörleri tek tek taramak yerine yakın komşu grafında arar
Sıra önemlidir: metadata hard filter → HNSW dense retrieval → cross-encoder rerank. Kesin şartları benzerlik skoruna bırakmıyoruz.
Cosine sezgisini tahtada anlatabilecek kadar açık tut. Bi-encoder hız sağlar çünkü doküman vektörleri cache'lenir. Cross-encoder daha doğru ama her query-document çiftini birlikte okuduğu için sadece top-N'de çalıştırılır.
6 Kendi eğittiğimiz NN — BGE-M3 + LoRA

Domain adaptation: genel model → Türkçe emlak

  • LoRA sezgisi: base'i DONDUR, yanına küçük adapter (B·A) ekle, sadece onu eğitΔW = (α/r)·B·A · r=8, α=16 · hedef: attention q/k/v × 24 katman
  • Param: ~1.18M eğitilebilir / ~560M toplam = %0.21 · adapter 4.7MB
  • Backprop: gradyan tüm ağdan akar, ama yalnız adapter güncellenir (optimizer sadece requires_grad)
Öğretme — triplet loss
Üçlüler: (sorgu, doğru ilan = pozitif, farklı ilan = negatif). Sorguyu pozitife yaklaştır, negatiften margin kadar uzaklaştır.
L = max(0, ‖q−p‖ − ‖q−n‖ + margin)
AdamW lr 2e-5 · 2 epoch · AMP mixed-precision · grad-accum 4 · mean-pool + L2-norm
TechPyTorch (CUDA)PEFT / LoRAtransformersRTX 5070 Ti
Full fine-tune yerine LoRA: param-verimli, küçük veride overfit düşük, base unutmaz. Backprop: kayıp gradyanı zincir kuralıyla geçer, donuk base üzerinden aktivasyon gradyanı akar ama ağırlık güncellemesi sadece adapter'da. Triplet: hizalama öğretir. [Detaylar çalışma kağıdında.]
6 LoRA backpropagation — ağırlıklar nasıl öğreniyor?

Base model sabit, gradyan adapter'a akar

W' = W + (α/r) · B · A

W: donuk base ağırlığı
A, B: eğitilen düşük-rank matrisler

Bizde `r=8`, `α=16`; attention katmanlarının query/key/value projeksiyonlarına adapter ekledik.

  • Forward pass: sorgu, pozitif ve negatif ilan embedding'leri hesaplanır
  • Triplet loss hangi çiftlerin fazla yakın / uzak olduğunu ölçer
  • Chain rule ile ∂L/∂A ve ∂L/∂B hesaplanır
  • AdamW yalnız adapter parametrelerini günceller; base ağırlığı W değişmez
Gradyanın ağdan geçmesi ile tüm ağırlıkların güncellenmesi aynı şey değildir. Base model üzerinden türev akar; optimizer yalnız `requires_grad=True` adapter parametrelerini değiştirir.
Quiz seviyesinde savunulacak slayt. Chain rule, frozen weights ve optimizer ayrımını vurgula. LoRA'nın amacı sadece hız değil: küçük veriyle tüm modeli bozma riskini azaltmak.
